Rendering the Buddhabrot with WebGL

27 Apr, 2018 · Read in about 6 min · (1115 Words)

Introduction

I finally decided to make my first web application. Being a c++ guy I want to use this little project to learn js,html etc, so if you encounter some stupid/noobish js usage/style pls let me know so I can improve.

If you don’t know what a Buddhbrot is I will give you a overview: The Buddhabrot is really cool mathematical object which somewhat looks like a Buddha:

live demo

To understand how a Buddhabrot is created one must first understand the famous Mandelbrot and the complex dynamic behind it. Both have their source in the complex recursive formula Z₁->Z₀+C. This harmless looking Formula results in some super complex behavior. Each starting point C has a unique orbit of Z values, here is the orbit for c = 0.5i:

As you can see the the Orbit of 0.5i is spiral-shaped and tends to a fixed point, the later is true for all orbits with a maximal length of 2. All orbits which at some point have a Z with |Z|>2 will go off to infinity. The Mandelbrot set now is the set of those C values that don’t tend to infinity and instead orbit around some fixed point.

The Buddhabrot now isn’t a set it’s a probability map. This map shows the probability for a point being crossed by the orbit of a point that is not in the Mandelbrot set. Bright parts have a higher probability of being in the path of a orbit then darker parts.

Implementation

I decided to use WebGL2 because I already know CoreGl and WebCL seems pretty dead. This created some inconveniences like the lack of compute shader and random memory access.

First I used GL’s transform feedback to generate some random points outside of the Mandelbrot set. To generate random points i first used the famous glsl random one liner: 

 
float rand(vec2 co,float seed){
    return fract(sin(dot(co.xy*seed ,vec2(12.9898,78.233))) * 43758.5453);
}
With a glVertexID plus a uniform random number as seed, but I later changed to xorshift because it has a better distribution. The seed-state is stored in a simple transform feedback loop.

To check weather or not a point is inside the set I used a pre rendered Mandelbrot mask texture with all pixels inside the set being black. Then I generated random points as long as the corresponding texture fetch returned 0. The mask texture: Here orbitSampleEnd and orbitSampleBegin are the lower and upper bounds of the Mandelbrot mask in world coords. 
vec2 pointOusideM()
{
    for(int i = 0;i<sampleCount;i++)
    {   
        vec2 uv =rand2();
        float I = texture(mandelmask,uv).r;
        if(I> 0)

             //Transform to world space

        }

This removes some precision but also yields a great deal of control and speed. The generated points a written via transform feedback in two different buffers. Next the a orbits are drawn and traced as points ,with additive blending to a floating point texture transform feedback to used to move the orbits via Z₁->Z₀+C. How many iterations this is done controls the level of detail.

uniform mat3 viewMatrix;
vec2 cpow2(vec2 z)
{
    return vec2(z.x*z.x-z.y*z.y,2.0*z.x*z.y);

    nextOrbitPosition = cpow2(orbitPosition)+orbitSource;

This creates a density map where points with high values are often visited by orbits. The last step is the normalization of the density map this is also done on the GPU with a cheap maximum reduction shader and two ping pong framebuffer to find the maximum value.

Importance Sampling

While the above works it’s way to slow and laggy. A great method to improve overall performance is to pick orbit starting points based on how long they will stay in the view range. To accomplish this a modified my Mandelbrot mask texture to indicate not only if point is outside the Mandelbrot set but also how «good» a point is. 

float budhhaImportance()
{
    vec2 p = uv*(orbitSampleEnd-orbitSampleBegin)+orbitSampleBegin;
    vec2 z = p;
    bool valid = false;
    float fitness = 0.0;
    
    for(int i;i<iterationCount;i++)
    {
       
        z = cpow2(z)+p;
        vec2 k = (viewMatrix*vec3(z,1)).xy;//View space
        if(abs(k.x)<1.0 && abs(k.y)<1.0)//Is in view range
        {
            fitness++;
        
         float d =dot(z,z);
        if(d>4.0)
        {
            valid = true;
        }
        if(d >12.0)
        {
            break;
        }
    }
    if(valid){
        return sqrt(fitness)/sqrt(float(iterationCount));//sqrt looks nicer
    }
    else
    {
        return 0.0;
    
}
Each iteration fitness is increased if z is in view range and then sqrt normalized. Here is what is looks like:

Looks pretty neat and show’s that the best orbits come from the border of the Mandelbrot set. Next I modified my orbit generation shader to evaluate a points importance.

vec2 pointOusideM()
{

    vec2 bestP = vec2(0);
    float bestI = 0.0;
    for(int i = 0;i<importanceSampleCount;i++)
    {   
        vec2 uv =rand2();
        float I = texture(importanceMap,uv).r;//importance
        if(I> bestI)
        {   
            bestP = uv*(orbitSampleEnd-orbitSampleBegin)+orbitSampleBegin;
            bestI = I;
        }
    }
    return bestP;
}

These modifications make rendering almost real time on my GTX 960m on a good desktop GPU it should be. GPU usage is also quiet good although I only checked it in Task manager. To further optimize the whole thing measuring would be a good idea because I am not sure what’s the main bottleneck. One good thing to try how ever would be to switch from float to half to reduce memory throughput.

The Js side of things

This was my first Java Script experience so I didn’t follow a specific style and improved while coding one of the first things I created was view matrix so the user can zoom around with smooth transitions. I didn’t know about prototypes thus its implemented like a c++ class. Later I discovered prototypes and the awesome js library community. So far js feels nice but a little washy. Anyway the actual Buddhabrot js class is really simple all GL objects a created and managed with twgl a really snazzy WebGL wrapper. There are two framebuffers one for the importance map and one for the orbit density then two buffers each for current orbit position,source,seed to enable a transform feedback loop. The importance map renderer and the reduction normalizer have their own separate classes. The gui was the hardest part until I discovered dat.gui which made everything easy and modular. If you are interested in the full yes implementation you can check it on git.
